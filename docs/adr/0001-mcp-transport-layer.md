# ADR-001: MCP トランスポート層の選定

## ステータス
承認

## コンテキスト
AI Agentがゲートウェイ経由でMCPサーバーにアクセスする際のトランスポート層を決定する必要がある。MCPプロトコルは stdio と HTTP/SSE の2つのトランスポートを公式にサポートしている。ゲートウェイはリモートのAI Agentから複数のリクエストを受け付けるため、ネットワーク越しに動作するトランスポートが必須となる。

## 決定
HTTP/SSE（Server-Sent Events）をMCPトランスポート層として採用する。

## 選択肢
| 選択肢 | 長所 | 短所 |
|--------|------|------|
| A: HTTP/SSE | ネットワーク越しに動作し複数クライアントを同時処理可能。既存のHTTPインフラ（ロードバランサー、TLS終端、認証ミドルウェア）をそのまま活用できる。リバースプロキシとして実装しやすい | WebSocketと比較して双方向通信が限定的。SSE接続の長時間維持にプロキシ設定の考慮が必要 |
| B: stdio | シンプルで遅延が小さい。MCP公式SDKのデフォルトサポート | プロセス間通信のためネットワーク越しに直接使用不可。1対1通信限定で複数クライアントの同時接続に対応できない。認証・認可レイヤーの挿入が困難 |

## 結果
ゲートウェイはHTTPリバースプロキシとして実装され、JSON-RPCリクエストのフォワーディングとSSEストリームの中継を担う。既存のHTTPミドルウェアパターン（Bearer トークン検証、監査ログ）を自然に統合でき、Go標準ライブラリの`net/http`で実装可能となった。
